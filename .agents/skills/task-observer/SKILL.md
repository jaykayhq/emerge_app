# Task Observer — Session Meta-Observability Skill

Monitors each work session to detect patterns the developer would improve if
they noticed them. Outputs structured observations and actionable recommendations
**for review**, not automatic fixes.

## Core Principle

> **Observe. Record. Recommend. Never edit without explicit approval.**

The developer is always in control. Task Observer surfaces what it found;
the developer decides what to do with it.

---

## Observation Loop

### 1. Session Start

When beginning any work session:

```
# Record session start
scripts/record_observation.py session_start \
  --session-id <auto-generated-or-from-context> \
  --task "<brief description of what you're about to do>"
```

If a previous session's observations are pending review, surface them first:

```
scripts/synthesize.py --review-mode
```

### 2. During the Session

Watch for these event types. Record each one **immediately** when it occurs.

| Event Type | Trigger | Example |
|---|---|---|
| `correction` | Developer pushes back on an approach | "No, use Riverpod not BLoC here" |
| `rework` | You undo/replace your own work | Discarded a provider, rewrote a DAO |
| `unmet_need` | User's true goal not achieved by delivered solution | User wanted debugging help, got code |
| `friction` | Something blocked progress | Build failed, wrong import, missing dependency |
| `pattern` | Recurring behavior noticed | Third time this session debugging the same error type |
| `success` | Something worked well | TDD caught the bug before it shipped |

Record each observation:

```bash
scripts/record_observation.py observe \
  --session-id <current-session-id> \
  --type correction \
  --description "User corrected navigation approach — need StatefulShellRoute for tab shell" \
  --evidence "Prompt: 'use StatefulShellRoute', my output used go_router redirects" \
  --severity 3
```

**Severity scale:**
- 1 = cosmetic (style, naming)
- 2 = minor (wasted time <5min, easily fixed)
- 3 = moderate (wrong approach, needed rework)
- 4 = significant (architecture mismatch, required redesign)
- 5 = critical (wrong feature entirely, security concern)

### 3. Session End

When finishing (or switching tasks):

```bash
scripts/record_observation.py session_end \
  --session-id <current-session-id>
```

Then synthesize recommendations:

```bash
scripts/synthesize.py --session-id <current-session-id> --output recommendations.json
```

Present the output to the developer as a **structured summary**, never silently
apply any recommendation.

## Output Format

### Raw Observations (JSONL)

Each line is a JSON object appended to `.agents/skills/task-observer/data/observations.jsonl`:

```json
{
  "id": "obs_<uuid>",
  "sessionId": "sess_<uuid>",
  "userId": "developer",
  "observationType": "correction|rework|unmet_need|friction|pattern|success",
  "description": "Plain text description of what happened",
  "evidence": "Supporting detail — prompt snippets, file paths, error messages",
  "severity": 1-5,
  "isResolved": false,
  "createdAt": "2026-07-06T14:32:00Z",
  "context": {
    "feature": "habits",
    "file": "lib/features/habits/presentation/providers/habit_providers.dart",
    "agent": "claude-code"
  }
}
```

### Recommendations (generated by synthesize.py)

```json
{
  "id": "rec_<uuid>",
  "observationId": "obs_<uuid>",
  "userId": "developer",
  "suggestion": "Add a SKILL.md rule: never use ref.watch inside go_router redirect — already in AGENTS.md but missed in new code",
  "rationale": "2 corrections this session about ref.watch in redirects",
  "basedOnPatterns": ["correction:ref.watch-redirect-001", "correction:ref.watch-redirect-002"],
  "priority": "high|medium|low",
  "createdAt": "2026-07-06T14:35:00Z",
  "isResolved": false
}
```

## Synthesize Outputs

After every session, present findings in this format:

```
## Task Observer — Session Summary

### Corrections (N)
| # | What was corrected | Lesson |
|---|---|---|
| 1 | ... | ... |

### Patterns Detected
- **Pattern**: Worked around X three times → Recommend formalizing as skill rule
- **Pattern**: Friction point Y recurred → Recommend adding to AGENTS.md gotchas

### Recommendations (presented for your review)
1. **[HIGH]** Suggestion... (based on N observations)
2. **[MEDIUM]** Suggestion...

No changes were auto-applied.
```

## Rules

1. **Never auto-edit.** Observations and recommendations are reports. The developer decides.
2. **Record immediately.** Don't batch observations — capture the moment.
3. **Be specific.** "Wrong routing pattern" is vague. "Used GoRoute widgets inside StatefulShellRoute branch — should mount branches as GoRoute children" is specific.
4. **Link to evidence.** Include file paths, prompt snippets, error messages.
5. **Respect existing rules.** If AGENTS.md already covers something, note "existing rule X not followed" rather than suggesting a duplicate.
6. **Compress patterns.** If the same correction type appears 3+ times across sessions, upgrade from observation to a skill-level recommendation.
7. **Resolve honestly.** When a recommendation is accepted, mark `isResolved: true`. When rejected, note why in context.

## Integration with Existing Tools

| Tool | Integration Point |
|---|---|
| `AGENTS.md` | Skill rules should reference existing AGENTS.md sections; don't duplicate |
| `.agents/skills/` | Recommendations may suggest new skill creation (use writing-skills process) |
| `scripts/dataset_distillation/` | Resolved patterns feed into seed data for fine-tuning |
| `AppLogger` (Flutter) | Not applicable — this is agent-side only |
| `EventBus` (Flutter) | Not applicable — this is agent-side only |

## Data Storage

All observations stored in agent-local JSONL at:
```
.agents/skills/task-observer/data/observations.jsonl
```

Recommendations stored at:
```
.agents/skills/task-observer/data/recommendations.jsonl
```

Session state tracked in:
```
.agents/skills/task-observer/data/sessions.jsonl
```

These are append-only. Never overwrite. Rotate by archiving rather than deleting.

---

## Cross-Skill Activation

- **writing-skills**: When a recommendation suggests a new skill rule
- **systematic-debugging**: When friction type is "build_failure" or "runtime_error"
- **test-driven-development**: When correction is about missing tests
- **verification-before-completion**: When pattern is "claimed done without running tests"
